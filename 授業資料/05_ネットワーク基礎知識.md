# ネットワーク基礎知識

> Webアプリ作成講座｜配布用教材 05

Webアプリは、ブラウザーだけで動いているわけではありません。端末、Wi-Fi、ルーター、DNS、Webサーバーなどが協力してデータを届けています。この教材では、Webページが表示される仕組みと、安全にネットワークを利用するための基本を学びます。

---

## 1. ネットワークとインターネット

**ネットワーク**とは、複数のコンピューターやスマートフォンなどをつなぎ、データをやり取りできるようにした仕組みです。

**インターネット**とは、世界中にある多くのネットワークが互いにつながった「ネットワークのネットワーク」です。

```text
家庭のネットワーク ─┐
学校のネットワーク ─┼─ インターネット ─ Webサーバー
会社のネットワーク ─┘
```

Webは、インターネットを利用するサービスの1つです。インターネットではWebのほかに、メール、オンラインゲーム、通話なども利用されています。

---

## 2. LAN・WAN・Wi-Fi

### LAN

**LAN（Local Area Network）**は、家庭、学校、会社など、管理された比較的限られた範囲のネットワークです。

```text
パソコン ─┐
スマートフォン ─ Wi-Fiアクセスポイント／ルーター ─ インターネット
ゲーム機 ─┘
```

### WAN

**WAN（Wide Area Network）**は、離れた場所のネットワーク同士を結ぶ広域ネットワークです。インターネットは多数のLANやWANなどが相互接続されたものです。

### Wi-Fi

**Wi-Fi**は、無線を使って端末をLANへ接続するための技術です。Wi-Fiそのものとインターネットは同じではありません。

- Wi-Fiへ接続できても、回線やルーターに問題があればインターネットは使えない
- インターネットにつながっていないWi-Fiネットワークも作れる
- 携帯電話回線では、Wi-Fiを使わずにインターネットへ接続できる

### SSIDとパスワード

- **SSID**：Wi-Fiネットワークを見分けるために表示される名前
- **パスワード**：接続を許可するための秘密情報
- **WPA2／WPA3**：Wi-Fi通信を保護するための方式

パスワードが設定されていても、そのWi-Fiや接続先のサイトが必ず安全とは限りません。知らないSSID、よく似た偽SSID、管理者のわからないネットワークには注意します。

---

## 3. ルーターとデータの通り道

**ルーター**は、異なるネットワークの間で、データをどちらへ送るか判断する機器や機能です。家庭用の機器には、Wi-Fiアクセスポイントや簡単なファイアウォールなど、複数の機能がまとめられていることがあります。

大きなデータは、小さなまとまりに分けて送られます。このまとまりを一般に **パケット** と呼びます。各パケットには、送り先や送り元など、配送に必要な情報が付いています。

```text
送るデータ
  ↓ 分割
[パケット1] [パケット2] [パケット3]
  ↓ ネットワークを通って配送
受信側で処理
```

実際の通信は、通信方式や層によってデータの呼び方や構造が異なります。この図は全体像をつかむための簡略化です。

---

## 4. IPアドレス

**IPアドレス**は、IP通信で送り先や送り元を識別するための番号です。

```text
IPv4の例：192.0.2.1
IPv6の例：2001:db8::1
```

これらは説明用として予約された例です。実際の接続先として使うものではありません。

### 「すべての端末が世界で1つのIPを持つ」とは限らない

IPアドレスは、使われる範囲や時間によって扱いが異なります。

- 家庭内だけで使う **プライベートIPアドレス** がある
- インターネット側から見える **グローバルIPアドレス** がある
- 接続のたびに変わるアドレスも、固定されるアドレスもある
- 1台の端末がIPv4とIPv6など複数のアドレスを持つことがある
- 家庭内の複数端末が、外部からは1つのIPv4アドレスを共有して見えることがある

家庭用ルーターなどが、内部のアドレスと外部のアドレスを変換する仕組みの1つを **NAT** といいます。

```text
家庭内
PC    192.168.1.10 ─┐
スマホ 192.168.1.11 ─┼─ ルーター／NAT ─ グローバルIPv4 ─ インターネット
ゲーム 192.168.1.12 ─┘
```

### IPv4とIPv6

| 種類 | 長さ | アドレス数 | 表記例 |
| --- | --- | --- | --- |
| IPv4 | 32ビット | 約43億 | `192.0.2.1` |
| IPv6 | 128ビット | 約3.4×10の38乗 | `2001:db8::1` |

IPv6は、IPv4アドレス不足への対応などを目的に設計されました。「無限」ではありませんが、非常に大きな数のアドレスを表せます。IPv4がなくなってIPv6へ一度に置き換わったわけではなく、現在は両方が利用されています。

## 5. ドメイン名とDNS

`example.com`のような、人が扱いやすい名前を **ドメイン名** といいます。

**DNS（Domain Name System）**は、ドメイン名に対応するIPアドレスなどの情報を調べる分散型の仕組みです。電話帳に例えられますが、世界に1冊だけの電話帳があるわけではありません。複数のDNSサーバーが協力し、結果を一時保存しながら答えます。

```text
1. ブラウザーに https://example.com/ を入力
2. 端末がDNSの情報を確認
3. 必要ならDNSサーバーへ問い合わせ
4. 対応するIPアドレスを得る
5. その接続先へ通信を始める
```

### 1つのドメイン＝1つの固定IPとは限らない

- 1つのドメイン名が複数のIPアドレスへ対応することがある
- 利用場所や時間によって異なるIPアドレスが返ることがある
- 複数のドメイン名が同じIPアドレスを使うことがある
- DNSにはメールサーバーなど、IPアドレス以外の情報も登録できる

大規模サービスでは、利用者に近いサーバーへ案内したり、負荷を分散したりするためにも使われます。

---

## 6. ポート番号

同じコンピューター上で複数のネットワークサービスを動かすため、通信では **ポート番号** も使われます。建物の住所がIPアドレス、受付窓口がポート番号、というイメージです。

| サービスの例 | よく使われるポート番号 |
| --- | ---: |
| HTTP | 80 |
| HTTPS | 443 |

URLで標準のポートを使う場合、通常は番号を省略します。

```text
https://example.com:443/
```

ポートが「物理的な穴」なのではなく、通信相手のサービスを識別するための番号です。

## 7. Webページが表示されるまで

ブラウザーへURLを入力してから表示されるまでを、簡単に見てみましょう。

```text
① URLを入力
   ↓
② DNSで接続先を調べる
   ↓
③ サーバーとの通信経路を作る
   ↓
④ HTTPSならTLSで相手を確認し、暗号化の準備をする
   ↓
⑤ ブラウザーがHTTPリクエストを送る
   ↓
⑥ サーバーがHTTPレスポンスを返す
   ↓
⑦ HTMLを読み、CSS・JavaScript・画像なども追加で要求する
   ↓
⑧ ブラウザーが内容を組み立てて表示する
```

実際にはキャッシュ、プロキシ、CDN、複数の通信方式なども関わりますが、まずはこの流れを理解しましょう。

## 8. HTTPのリクエストとレスポンス

**HTTP（Hypertext Transfer Protocol）**は、Webでクライアントとサーバーが情報をやり取りするための決まりです。

### リクエスト

ブラウザーなどがサーバーへ送る要求です。

```http
GET /profile HTTP/1.1
Host: example.com
```

おもなHTTPメソッド：

| メソッド | おもな目的 |
| --- | --- |
| `GET` | 情報を取得する |
| `POST` | データを送り、新しい処理や作成を依頼する |
| `PUT` | リソース全体を置き換える |
| `PATCH` | リソースの一部を変更する |
| `DELETE` | リソースを削除する |

実際に何をするかは、Webアプリ側の設計によって決まります。メソッド名だけで権限や安全性が保証されるわけではありません。

### レスポンス

サーバーが返す応答です。

```http
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8

<!doctype html>
...
```

レスポンスには、処理結果を示すステータスコード、追加情報を持つヘッダー、ページなどの本文が含まれます。

---

## 9. HTTPステータスコード

HTTPステータスコードは3桁の数字です。最初の数字で大きな分類がわかります。

| 範囲 | 分類 | 意味 |
| --- | --- | --- |
| 100〜199 | 情報 | 処理を続けているなど |
| 200〜299 | 成功 | リクエストを受け付けた |
| 300〜399 | リダイレクト | 別の場所やキャッシュを利用する |
| 400〜499 | クライアントエラー | リクエストを処理できない |
| 500〜599 | サーバーエラー | サーバー側で処理に失敗した |

### よく見るコード

| コード | 名前 | 意味の例 |
| ---: | --- | --- |
| 200 | OK | リクエストが成功した |
| 201 | Created | 新しいリソースを作成した |
| 204 | No Content | 成功したが返す本文はない |
| 301 | Moved Permanently | 恒久的に別のURLへ移動した |
| 302 | Found | 一時的に別のURLへ案内する |
| 304 | Not Modified | キャッシュ済みの内容を再利用できる |
| 400 | Bad Request | リクエストの形式などに問題がある |
| 401 | Unauthorized | 認証が必要、または認証に失敗した |
| 403 | Forbidden | サーバーは理解したがアクセスを許可しない |
| 404 | Not Found | 指定したリソースが見つからない |
| 405 | Method Not Allowed | そのHTTPメソッドを受け付けない |
| 429 | Too Many Requests | 短時間に要求を送りすぎた |
| 500 | Internal Server Error | サーバー内部で予期しない問題が起きた |
| 502 | Bad Gateway | 中継先から正しい応答を得られない |
| 503 | Service Unavailable | 一時的にサービスを利用できない |

### コードから原因を決めつけない

`500`はサーバー側の失敗を表しますが、Pythonの文法エラーだけが原因ではありません。設定、データベース、外部サービスなど、多くの原因が考えられます。ステータスコードは調査の出発点です。

## 10. HTTPとHTTPS

**HTTPS**は、HTTPの通信を **TLS** で保護します。古い名称のSSLという言葉が残って「SSL/TLS」と呼ばれることもありますが、現在のHTTPSではTLSが使われます。

HTTPSがおもに提供するもの：

- **暗号化**：通信途中で内容を読まれにくくする
- **完全性**：通信途中の書き換えを検出できるようにする
- **認証**：証明書を使い、接続先ドメインとの通信であることを確認する

### HTTPSでもサイトの内容が安全とは限らない

HTTPSは **通信経路** を保護します。次のことまでは保証しません。

- 運営者が信用できる
- 商品や情報が正しい
- 詐欺サイトやフィッシングサイトではない
- ダウンロードしたファイルに問題がない
- 端末自体がマルウェアに感染していない

偽サイトもHTTPSを利用できます。アドレスバーでは、鍵の絵だけでなく **ドメイン名が正しいか** を確認します。ブラウザーによって表示方法は異なります。

## 11. プロキシ・リバースプロキシ・CDN

### プロキシ

クライアント側の代わりに外部へアクセスする中継役を、一般にフォワードプロキシと呼びます。組織内のアクセス制御などに使われます。

### リバースプロキシ

Webサーバー側の手前でリクエストを受け、適切なサーバーへ転送する仕組みです。

```text
利用者 → リバースプロキシ → WebアプリA
                       ├→ WebアプリB
                       └→ WebアプリC
```

おもな用途：

- TLS通信の処理
- 複数サーバーへの負荷分散
- キャッシュ
- URLに応じた転送
- ログの記録
- 一部の不正な通信を制限する

リバースプロキシを置くだけでWebアプリが安全になるわけではありません。アプリの認証、入力処理、更新、アクセス権限なども必要です。

### CDN

**CDN（Content Delivery Network）**は、各地のサーバーなどを利用し、画像やCSSなどを利用者へ効率よく届ける仕組みです。配信の高速化、元サーバーの負荷軽減、可用性向上などに使われます。

---

## 12. ファイアウォールとVPN

### ファイアウォール

ファイアウォールは、決められたルールに基づいてネットワーク通信を許可または拒否します。IPアドレス、ポート番号、通信の状態、アプリケーション情報などを判断に使う製品があります。

```text
外部ネットワーク → [ファイアウォールのルール] → 内部ネットワーク
```

「不審な通信をすべて自動で見破る壁」ではありません。許可された通信を悪用する攻撃、設定ミス、内部からの問題などは、ファイアウォールだけでは防げません。

### VPN

**VPN（Virtual Private Network）**は、離れた端末やネットワークの間に、暗号化などで保護した論理的な通信経路を作る技術です。会社や学校の内部ネットワークへ遠隔接続する用途や、VPN事業者を経由してインターネットへ接続する用途などがあります。

VPNを使うと、端末からVPN接続先までの通信を保護できます。一方で、次の点に注意が必要です。

- VPN接続先より先は別の区間になる
- VPN事業者を信頼する必要がある
- フィッシングやマルウェアを自動で防ぐものではない
- HTTPSで保護されている通信の安全性が、VPNなしですべて失われるわけではない
- 学校や家庭のルール、利用規約に従う必要がある

「VPNを使えば何をしても安全」ではありません。

## 13. ネットワーク上の脅威

この章の目的は攻撃方法を試すことではなく、仕組みを知って防ぐことです。許可のないシステムへ侵入したり、負荷をかけたり、他人の情報を取得したりしてはいけません。

### フィッシング

本物の企業や知人を装い、偽サイトなどへ誘導してパスワードや個人情報を入力させる手口です。

よくある特徴：

- 「今すぐ」「停止されます」など急がせる
- パスワードや確認コードを求める
- 表示文字と実際のリンク先が違う
- ドメイン名が本物と少しだけ違う
- 予想していない添付ファイルがある

不安なときはメッセージ内のリンクを使わず、公式アプリや自分のブックマークから確認します。家族、先生、サービスの正式な窓口へ相談しましょう。

### マルウェア

利用者やシステムへ害を与える目的で作られたソフトウェアの総称です。

- **ウイルス**：ほかのファイルなどへ入り込み、実行をきっかけに広がるもの
- **ワーム**：ネットワークなどを通じて自ら広がるもの
- **トロイの木馬**：正規のアプリなどを装うもの
- **ランサムウェア**：データを暗号化するなどして身代金を要求するもの
- **スパイウェア**：利用者の情報を秘密に収集するもの

実際のマルウェアは複数の特徴を持つことがあります。

### DDoS攻撃

多数の機器などから大量の通信を送り、サービスを利用しにくくする攻撃です。自分の端末がマルウェアに感染し、本人が気づかないまま攻撃へ使われることもあります。

### SQLインジェクション

利用者の入力をSQL文へ安全でない方法で連結したWebアプリに対し、入力をデータではなく命令の一部として解釈させる攻撃です。

開発側のおもな対策：

- 文字列連結でSQLを組み立てない
- パラメーター化クエリーやORMを正しく使う
- データベース権限を必要最小限にする
- エラー内容を利用者へ詳しく出しすぎない
- 入力の形式や長さも確認する

「危険な文字だけを削除する」方法や、入力チェックだけでは十分ではありません。

## 14. 今日からできる安全対策

### アカウント

- パスワードは長くし、サービスごとに違うものを使う
- 信頼できるパスワードマネージャーの利用を検討する
- 利用できるサービスでは多要素認証（MFA）を有効にする
- MFAの確認コードを他人へ教えない
- 身に覚えのないログイン通知を確認する

### 端末とアプリ

- OS、ブラウザー、アプリを更新する
- 公式または信頼できる配布元から入手する
- 不要なアプリやブラウザー拡張機能を削除する
- 画面ロックと端末の紛失対策を設定する
- 大切なデータはバックアップする

### メール・メッセージ・Web

- 急がせる内容でも、すぐリンクを押さない
- ドメイン名と送信元を確認する
- パスワードやMFAコードを送らない
- 不審な添付ファイルを開かない
- 困ったら1人で判断せず、保護者や先生へ相談する

### 公共のネットワーク

- 管理者がわかる正しいSSIDか確認する
- HTTPSと正しいドメイン名を確認する
- 共有設定や自動接続を必要に応じて無効にする
- 重要な操作は信頼できる回線へ切り替えることも検討する
- VPNを使う場合も、提供者と利用規約を確認する

安全対策はリスクを減らすもので、1つの対策だけで完全に安全になるわけではありません。

---

## 15. 開発者として守ること

Webアプリを作る側には、利用者の情報を守る責任があります。

- HTTPSを使用する
- パスワードを平文で保存しない
- 秘密鍵やAPIキーをHTMLや公開リポジトリへ置かない
- サーバー側でも入力を検証する
- データベース操作にパラメーター化クエリーや安全なORMを使う
- 利用者と管理者の権限を分ける
- エラーページに秘密情報や詳細な内部情報を出さない
- フレームワークやライブラリーを更新する
- 必要以上の個人情報を集めない
- 問題に気づけるようログを取り、秘密情報はログへ残さない

クライアント側のHTML属性やJavaScriptだけでは、攻撃者によるリクエストを防げません。重要な確認とアクセス制御はサーバー側でも行います。

## まとめ

- インターネットは、多数のネットワークが相互接続された仕組み
- Wi-Fiは無線LANの接続技術であり、インターネットそのものではない
- IPアドレスは通信相手を識別し、DNSはドメイン名に対応する情報を調べる
- WebではHTTPリクエストとHTTPレスポンスをやり取りする
- ステータスコードは処理結果の分類を示し、原因調査の手がかりになる
- HTTPSは通信経路を保護するが、サイト自体の信頼性までは保証しない
- ファイアウォール、VPN、リバースプロキシは役立つが、1つですべてを防げない
- 長く固有のパスワード、MFA、更新、フィッシングへの注意を組み合わせる
- Webアプリでは、入力値や権限をサーバー側でも安全に扱う
