# Webページ制作の流れ

> Webアプリ作成講座｜配布用教材 02

Webページは、いきなりコードを書き始めるよりも、目的や内容を整理してから作るほうが、わかりやすく使いやすいものになります。この教材では、企画から公開・改善までの流れを学びます。

---

## 1. 制作の全体像

この講座では、Webページを次の8段階で制作します。

```text
1. 企画する
      ↓
2. 内容を集めて整理する
      ↓
3. サイトマップを作る
      ↓
4. ワイヤーフレームを作る
      ↓
5. デザインする
      ↓
6. コーディングする
      ↓
7. テストする
      ↓
8. 公開して改善する
```

実際の制作では、前の段階へ戻って考え直すこともあります。たとえば、コーディング中に情報が足りないと気づいたら、企画や内容の整理に戻ります。

---

## 2. 企画する

企画では「何を作るか」だけでなく、「なぜ作るか」「だれが使うか」を決めます。

### 目的を決める

まず、Webページを作る目的を1文で書きます。

悪い例：

> 部活動のサイトを作る。

よい例：

> 新入生に活動内容と見学方法を伝え、安心して見学を申し込めるようにする。

「見た人に、最後にどうなってほしいか」まで考えると、必要な情報や機能を決めやすくなります。

### 対象者を考える

サイトを利用してほしい人を **ターゲットユーザー** といいます。年齢や性別だけで決めるのではなく、次の点を考えましょう。

- どのような目的でサイトを見るか
- いつ、どこで、どの端末から見るか
- 何を知りたいか
- 何に困っているか
- Webサイトの操作にどのくらい慣れているか

### ペルソナとは

ペルソナは、ターゲットユーザーを考えやすくするために作る、具体的な利用者像です。実在する個人の詳しい情報を書くものではありません。必要以上に決めつけず、制作に必要な特徴だけを設定します。

例：

| 項目 | 内容 |
| --- | --- |
| 名前 | 佐藤さん（架空） |
| 状況 | 入学したばかりの中学1年生 |
| 見る場面 | 帰宅後、スマートフォンで見る |
| 知りたいこと | 活動日、必要な道具、見学方法 |
| 困りごと | 知らない先輩へ直接質問するのは緊張する |

### 成功の条件を決める

完成したかどうかを判断できる条件も決めます。

- 活動日と場所をすぐ見つけられる
- スマートフォンでも横にはみ出さない
- 見学方法が中学生にもわかる文章で書かれている
- すべてのリンクが正しく動く

### やってみよう：企画メモ

次の空欄を埋めましょう。

- 作るもの：
- 作る目的：
- 対象者：
- 対象者が知りたいこと：
- 見た後にしてほしいこと：
- 完成したと判断する条件：

---

## 3. 内容を集めて整理する

デザインの前に、掲載する文章、写真、ロゴ、リンクなどを集めます。内容が決まっていないまま画面を設計すると、後から情報が入りきらなくなることがあります。

### 情報に優先順位を付ける

集めた情報を3段階に分けてみましょう。

| 優先度 | 意味 | 例 |
| --- | --- | --- |
| 高 | 目的を達成するために必須 | 活動日、場所、申し込み方法 |
| 中 | あると理解しやすい | 活動写真、よくある質問 |
| 低 | なくても目的は達成できる | 過去の細かな記録 |

重要な情報ほど、見つけやすい場所へ置きます。

### 素材を使うときの注意

- 他人が作った文章や画像を、許可なくコピーしない
- 素材の利用条件と、クレジット表記が必要かを確認する
- 人物が写った写真は、公開してよいか確認する
- 本名、住所、電話番号、現在地などの個人情報を安易に公開しない
- 内容が正しいか、情報源や更新日を確認する

---

## 4. サイトマップを作る

サイトマップは、Webサイトに必要なページと、ページ同士のつながりを表した設計図です。

たとえば、部活動サイトなら次のように整理できます。

```text
ホーム
├─ 活動紹介
│  ├─ 活動内容
│  └─ 年間予定
├─ メンバー・作品
├─ よくある質問
└─ 見学案内
```

### サイトマップを作る手順

1. 必要な情報を付箋やメモに1つずつ書く
2. 似た内容をグループにする
3. グループごとにページ名を付ける
4. 最初に開くホームページを決める
5. ページ間をどう移動するか線で結ぶ
6. 不要なページや、足りないページがないか確認する

### 1ページだけの場合

1枚のWebページだけを作る場合も、情報の順序を考えます。

```text
ページタイトル
↓
短い紹介
↓
くわしい説明
↓
写真や作品
↓
申し込み・次の行動
```

---

## 5. ワイヤーフレームを作る

ワイヤーフレームは、各ページの「何を、どこに置くか」を示す画面の設計図です。この段階では色や細かな飾りよりも、情報の順序と使いやすさを考えます。

```text
┌────────────────────────┐
│ ロゴ             メニュー │
├────────────────────────┤
│                        │
│     ページの見出し       │
│     短い説明文           │
│     [見学方法を見る]      │
│                        │
├───────────┬────────────┤
│ 活動写真    │ 活動内容     │
├───────────┴────────────┤
│ よくある質問             │
├────────────────────────┤
│ フッター                 │
└────────────────────────┘
```

### ワイヤーフレームで確認すること

- いちばん大切な情報が目に入りやすいか
- 見出しと本文の順序が自然か
- ボタンを押した後に何が起こるかわかるか
- メニューから目的のページへ移動できるか
- 内容が多すぎて読みにくくなっていないか

### レスポンシブデザイン

画面の幅に合わせてレイアウトを変え、パソコンやスマートフォンなどで使いやすくする考え方を **レスポンシブデザイン** といいます。

パソコン版を小さく縮めるだけでは、文字やボタンが小さすぎることがあります。少なくとも、広い画面と狭い画面のワイヤーフレームを考えましょう。

```text
パソコン：画像 │ 説明文
スマートフォン：画像
                 ↓
               説明文
```

### やってみよう

紙と鉛筆を使い、企画したページのスマートフォン版ワイヤーフレームを作りましょう。四角形と文字だけでかまいません。

---

## 6. デザインする

ワイヤーフレームをもとに、色、文字、画像、余白、ボタンの形などを決めます。完成画面に近い見本を **デザインカンプ** や **モックアップ** と呼びます。

### 最初に決めること

- メインカラー、サブカラー、背景色
- 見出しと本文の文字サイズ
- 写真やイラストの雰囲気
- ボタンやリンクの見た目
- 要素どうしの余白

### デザインのチェックポイント

- 目的と対象者に合った雰囲気か
- 文字と背景を十分に見分けられるか
- 色だけで情報を区別していないか
- 同じ役割の部品は同じ見た目になっているか
- どこを押せるかひと目でわかるか
- 文字を大きくしても内容を読めるか

参考サイトのデザインを観察するのはよい方法ですが、そのままコピーせず、「なぜ見やすいのか」を考えて自分の制作に生かしましょう。

---

## 7. コーディングする

設計とデザインを、ブラウザーが読み取れるファイルにします。

| 種類 | おもな役割 | 代表的なファイル名 |
| --- | --- | --- |
| HTML | 内容と構造を表す | `index.html` |
| CSS | 色、文字、配置などを整える | `style.css` |
| JavaScript | 操作に応じた処理や動きを加える | `script.js` |
| 画像など | 写真、イラスト、動画、音声 | `logo.png` など |

画像や動画はコードではありません。元教材の「ソースは主に4種類」という表現は避け、この教材では「制作に使うファイル」として分けています。

### フォルダー構成の例

```text
my-site/
├─ index.html
├─ css/
│  └─ style.css
├─ js/
│  └─ script.js
└─ images/
   └─ activity.jpg
```

### コーディング中のポイント

- 小さな部分を作るたびにブラウザーで確認する
- ファイル名は内容がわかる半角英数字にする
- 同じ名前のファイルを何個も作らない
- HTMLでは見た目だけでなく内容に合う要素を使う
- 他の人が読んでもわかるように、字下げや命名をそろえる
- 変更前の状態へ戻せるよう、こまめに保存する

---

## 8. テストする

自分の画面で表示できただけでは完成ではありません。公開前に、対象者が目的を達成できるか確認します。

### おもなテスト

| テスト | 確認すること |
| --- | --- |
| 内容 | 誤字、古い情報、説明不足がないか |
| 操作 | リンクやボタンが正しく動くか |
| 表示 | 文字、画像、レイアウトが崩れていないか |
| 端末・画面幅 | パソコンやスマートフォンで使えるか |
| アクセシビリティ | キーボード操作、代替テキスト、色の見分けやすさなど |
| 性能 | 画像が大きすぎず、表示を長く待たせないか |
| 安全・プライバシー | 個人情報や秘密の値が含まれていないか |

### 人に使ってもらう

制作した本人はページの場所を覚えているため、問題に気づきにくいことがあります。友達に説明せず使ってもらい、次の点を観察しましょう。

- 最初にどこを見るか
- 目的の情報を見つけられるか
- どこで迷うか
- 押せる場所を正しく判断できるか

テストする人が迷ったときは、その人の間違いではなく、改善の手がかりだと考えます。

---

## 9. Webに公開する

作ったファイルを自分のパソコンで開いただけでは、ほかの人はインターネットから閲覧できません。公開用のコンピューターやサービスへファイルを置き、URLからアクセスできる状態にすることを **デプロイ** といいます。

### 公開方法

| 方法 | 特徴 |
| --- | --- |
| レンタルサーバー | サーバーの保存場所を借り、管理画面やFTPなどでファイルを置く |
| 静的サイトホスティング | HTML・CSS・JavaScriptなどを手軽に公開できる |
| Git連携 | GitHubなどへ変更を送ると、自動で公開内容を更新できる |
| クラウドサービス | サーバー側の処理やデータベースを使うWebアプリにも対応できる |

公開するだけなら、自分でサーバー機器を購入して一から構築する必要はありません。サービスによって無料枠、料金、年齢制限、公開範囲、利用規約が異なります。授業では講師の指示に従ってください。

### ドメインとは

`example.com` のような、インターネット上の接続先を表す名前をドメイン名といいます。登録できるドメイン名は同じ範囲内で重複しないよう管理され、通常は一定期間ごとに利用料金を支払って更新します。

ホスティングサービスから `username.github.io` や `project.pages.dev` のようなURLが提供される場合は、最初から独自ドメインを取得しなくても公開できます。

### FTPとは

FTPはサーバーとファイルをやり取りするための通信方法です。レンタルサーバーではFTPソフトを使うことがありますが、公開方法はFTPだけではありません。管理画面へのアップロード、専用コマンド、Git連携などを使うサービスもあります。

### 公開前の安全確認

- 本名、住所、学校名、連絡先、位置情報を公開してよいか確認した
- 他人の写真や作品について許可を得た
- APIキー、パスワード、秘密の設定値をファイルに書いていない
- 問い合わせ先や投稿機能がある場合、情報の扱い方を決めた
- 利用するサービスの規約と公開範囲を確認した
- 必要に応じて保護者や講師の確認を受けた

---

## 10. 公開後も改善する

公開はゴールではなく、運用の始まりです。

```text
公開 → 使ってもらう → 意見や問題を集める → 直す → 再公開
```

次のような作業を続けます。

- 古くなった日時や文章を更新する
- 切れたリンクを直す
- 見つかった不具合や読みにくい部分を改善する
- 不要になった個人情報を削除する
- ファイルや設定のバックアップを取る
- 更新内容と日付を記録する

---

## まとめ

- コードを書く前に、目的・対象者・内容を整理する
- サイトマップでページ構成、ワイヤーフレームで画面構成を決める
- デザイン後にHTML、CSS、JavaScriptなどで形にする
- 公開前に、内容・操作・表示・安全性をテストする
- 公開には複数の方法があり、FTPだけが選択肢ではない
- 公開後も情報を更新し、利用者の意見をもとに改善する
